# replay.html 設計仕様書 — 走行ログ再生（OSM＋GPS＋パワーフロー）

> ロギングした走行データ（GPS＋OBD）を OpenStreetMap 上で再生する新規ページ。
> 既存 `viewer.html` とは別ファイル `replay.html` として作る。
> ステータス: 設計確定（2026-07。実装はこれから）。

---

## 1. 目的

fl4obd で記録した1回の走行を、地図・自車位置・パワーフローで同期再生する。「走行コースの再現」と「その時の車両状態」を1画面で追える再生ビューア。

---

## 2. 入力（MUST: 2ファイル）

| ファイル | 内容 | 用途 |
|---|---|---|
| `fl4obd_*.gpx` | GPSトラック（`<trkpt lat lon><time>`, 時刻=UTC ISO, ~1Hz） | 自車位置・経路 |
| `fl4obd_*.txt` | フレームログ（生 `622920` 応答, ~2Hz） | **再パースで全トルク・全電力**（peng/pgen/pdrive/pbat/mode/rpm/vsp/soc 等） |

- 物理値CSV(`_data_.csv`)ではなく **フレーム.txt を必須**とする。理由：パワーフローに必要な `pgen`/`pdrive` は CSV に無く、フレーム再パースでしか全項目が揃わない。
- 2ファイルを読み込み、**時刻（UTC ISO）で自動結合**する。
- パースは `index.html` の `parse22` 相当を流用（オフセット・スケールは同一）。

---

## 3. 画面構成（3タイル・ダッシュボード）

3つのタイルを**1画面に同時表示**。タブ切替ではない。1本の再生クロックで全タイル同期。

### 画面1：俯瞰（経路全体）
- 全経路を `fitBounds` で**固定表示**。再生中はパン/ズームしない。
- マーカー：**始点・終点（固定）**、**現在地（移動）**。
- **過去経路＝明色、未来経路＝暗色**（単純な明暗。データ色分けはしない）。再生進行で明色部が伸びる。
- マップ固定・自車が動く。

### 画面2：追従ズーム
- 拡大表示。**自車を中心に固定し、マップが動く**（毎フレーム `setView(現在地)`）。
- **ズーム可変**（UI/ピンチ）。
- **ノースアップ**（北固定＝Leaflet 標準。進行方向アップ＝マップ回転は採用しない）。

### 画面3：パワーフロー
- 既存 FLOW（エネルギーフロー SVG）を流用。時刻 t の BAT/GEN/ENG/TRC＋車輪の電力フローを描画。
- `index.html` の parse22→`D.peng/pgen/pdrive/pbat` と flow 描画ロジックを移植・共有。

---

## 4. レイアウト（レスポンシブ）

要素配置は画面比で変わるが、**表示内容は不変**。

### 縦（Pixel 10 基準, ≈9:20）
```
┌───────────────┐
│   画面2 追従ズーム │  上部フル幅
├───────┬───────┤
│画面1 俯瞰│画面3 フロー│  下段 左|右
└───────┴───────┘
[ ▶ ━━━●━━━ 1x ]   再生バー(全幅)
```

### 横（PC 16:9）— 画面2を主役・左大
```
┌──────────────┬───────┐
│              │ 画面1  │
│   画面2       │ 俯瞰   │
│  追従ズーム    ├───────┤
│              │ 画面3  │
│              │ フロー  │
└──────────────┴───────┘
[ ▶ ━━━━━━●━━━━ 1x ]
```

- 縦/横は画面比（`orientation` / アスペクト）で自動切替。
- 各タイルは独立にリサイズ可能な領域（Leaflet map は `invalidateSize` を呼ぶ）。

---

## 5. 再生エンジン

- **単一の再生クロック**（現在時刻 t, UTC）で3タイルを同期描画。
- コントロール（全幅の再生バー）：
  - 再生 / 停止
  - シーク（タイムライン・スクラブ）
  - 速度（1x / 2x / 5x / 10x）
- 既存 FLOW の再生機構（`FL.vt` / `FL.speed` / `flowRenderAt(t)` の系）を流用・一般化する。
- 地図タイルに小さな **HUD**（速度・モード等）を重畳。

---

## 6. 時刻同期

全ソースを **UTC ISO8601** に正規化して結合する。

| ソース | 時刻の持ち方 | 絶対時刻の作り方 |
|---|---|---|
| GPX | `<time>` = UTC ISO（`toISOString()`） | そのまま |
| フレーム.txt | 各行 `[HH:MM:SS.mmm]`（UTC。`toISOString().slice(11,23)` 由来）＋ヘッダ `Generated: <ISO>` | **ヘッダの日付 ＋ 行の時刻** で絶対UTCを復元 |
| 物理値.csv（使う場合） | 各行に完全 ISO（`toISOString()`） | そのまま |

- 結合：**最近傍タイムスタンプ**。GPS(~1Hz)は位置を内挿、OBD(~2Hz)は最近傍。
- **限界（軽微）**：フレーム.txt の行時刻は時分秒のみのため、走行が **UTC 日跨ぎ**だと日付復元に注意（時刻が巻き戻ったら日付+1）。通常の走行時間なら問題なし。
  - 恒久策：将来フレームログの各行にも完全 ISO を持たせれば解消（現状は viewer 互換のため据え置き）。

---

## 7. 位置の内挿

### Phase 1（初手・必須）
- **時間線形内挿**：GPS fix 2点間を時刻比で内挿し、自車マーカーを滑らかに動かす。
- **車速クロスチェック（任意）**：`OBD車速の積分 ≒ GPS 2点間距離` を検証し、GPS ジャンプ/トンネルを検出。

### Phase 2（後付け・精度オプション）
- **マップマッチング＋車速の弧長積分**：
  1. GPSトラックを OSM 道路網にマッチング（**OSRM `/match`** 丸投げが本命。DIY は Overpass で `highway=*` 取得＋最近傍スナップ、単純路限定）。
  2. マッチした道路が**経路形状（カーブ）**を与える。
  3. 車速を弧長 `s(t)=s(t0)+∫v dt` で積分し、道路ジオメトリ上へ写像 → 道なりの高レート位置。
- 効果：**画面2の高ズームで角切りが道なりに滑らか**に。俯瞰(画面1)は生1Hzで十分。
- 誤差源：交差点の分岐選択・並走路/立体交差/ロータリーの誤マッチ・未整備路の破綻（生GPSへフォールバック）。

---

## 8. 技術スタック

- **地図**：Leaflet + OSM タイル（`tile.openstreetmap.org`、個人利用範囲）。
  - map インスタンス2枚：画面1（固定 `fitBounds`）／画面2（`setView` 追従）。同一トラックを描画。
- **フロー**：既存 SVG＋parse22 を移植（`index.html` と共有）。
- **依存**：Leaflet（CDN or 同梱）。Phase 2 で OSRM `/match`（外部API）。
- **新規ファイル**：`replay.html`（単一HTML、`index.html`/`viewer.html` と並列）。

---

## 9. 必要データ項目（フレーム再パースから）

`index.html` の parse22 と同じ抽出：
- `mode`(off77)、`eng_rpm`(off98)、`vsp`(off13/spd_echo)、`batt_pwr`(off84)、`drv_trq3`(off88)、`gen_trq`(off90)、`eng_trq`(off94)
- 算出：`peng=2.09e-6·eng_trq·rpm`、`pgen=3.92e-6·|gen_trq|·rpm`、`pdrive=0.000147·drv_trq3·vsp`、`psys`、`pbat`、`soc` 等。

---

## 10. 実装フェーズ

1. **Phase 1（骨格）**：2ファイル読み込み＋時刻結合／3タイル・レスポンシブ／地図2枚（画面1固定・画面2追従）／再生同期／画面3フロー流用／時間線形内挿。
2. **Phase 2（精度）**：マップマッチング＋車速弧長積分（OSRM `/match`）。車速クロスチェック。

---

## 11. 制約・前提

- **実機は HTTPS 必須**（GitHub Pages）。GPS/Web Bluetooth とも secure context 前提。
- GPS は **~1Hz が実質上限**（ブラウザ Geolocation にレート指定なし）。密度は内挿で補う。
- OSM タイル・OSRM public API は**個人利用の範囲**。多用途化するならタイル/マッチのプロバイダを検討。
- Phase 2 のマップマッチングは複雑路で誤マッチしうる（§7 誤差源）。生GPSを常にフォールバックに保持。
